## Context

This is Phase 3 of the i18n multilingual initiative. Phases 1+2 shipped successfully for public-website (205 strings) and karaoke-decide (811 frontend + 47 backend strings). The patterns are proven â€” we're replicating them in karaoke-gen, the most complex project (many frontend surfaces + multi-layered backend).

## Scope Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Admin pages (`/admin/*`) | **Exclude** from i18n | Internal-only, ~30% more strings for zero user impact. Can add later. |
| Tenant default locale | **Out of scope** | Orthogonal to user locale; user's localStorage/browser detection suffices |
| Hash routing compatibility | **No conflict** | Hash (`#/jobId/review`) is client-side only, preserved across locale changes |
| Providers in root layout | **Keep in root** | Unlike Phase 1/2, karaoke-gen has admin pages outside `[locale]` that need ThemeProvider/TenantProvider. Only NextIntlClientProvider moves to `[locale]/layout.tsx`. |

## Architecture: Frontend Key Difference from Phase 1/2

karaoke-gen has admin pages at `app/admin/` that must NOT be under `[locale]` routing. So:

- **Root `app/layout.tsx`**: Keeps `<html>`, `<body>`, ThemeProvider, TenantProvider, ImpersonationBannerWrapper, ServiceWorkerRegistration, GoogleAnalytics. Remove hardcoded `lang="en"`.
- **`app/[locale]/layout.tsx`**: Adds NextIntlClientProvider + inline `<script>` to set `document.documentElement.lang`. Uses `generateStaticParams` for locale pre-rendering.
- **`app/admin/layout.tsx`**: Unchanged â€” inherits root providers, no locale context needed.

## Key Technical Details

### Accept-Language Header (frontend â†’ backend)
```typescript
// frontend/lib/api.ts
function getLocaleFromPath(): string {
  if (typeof window === 'undefined') return 'en';
  const match = window.location.pathname.match(/^\/(en|es|de)\//);
  return match ? match[1] : 'en';
}
```
Added to `getAuthHeaders()` so every API call includes locale.

### User Locale Persistence
- **Frontend:** `localStorage.setItem('locale', newLocale)` on language switch
- **Backend:** `User.locale` field in Firestore (`gen_users` collection), set on magic link verify from `Accept-Language`, updated on explicit language change
- **Emails/push:** Read `User.locale` from Firestore, fallback to `'en'`

### Static Export with Locale Routing
`generateStaticParams` on `[locale]/layout.tsx` returns `[{locale:'en'},{locale:'es'},{locale:'de'}]`. Combined with `output: 'export'` and `trailingSlash: true`, Next.js generates `/en/`, `/es/`, `/de/` directories â€” compatible with Cloudflare Pages.

### Backend-Generated URLs
All backend URLs that point to frontend must include locale prefix:
```python
f"{frontend_url}/{get_locale_prefix(user_locale)}/app/jobs#{job_id}/review"
```
